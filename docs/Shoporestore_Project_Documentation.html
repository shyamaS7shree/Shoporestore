01 · Overview

Executive overview

Shoporestore is a full-stack shopping platform built to demonstrate a complete customer journey: discovering products, filtering a catalog, viewing product details, maintaining a bag and wishlist, authenticating an account, saving addresses, paying or selecting cash on delivery, viewing orders, cancelling orders, requesting returns, reviewing purchased products, and contacting support.

The user interface is a Next.js App Router application. It communicates through a shared API bridge with a separate ASP.NET Core Web API. The backend uses Entity Framework Core with Microsoft SQL Server and contains compatibility logic for the existing Shopore_DB legacy schema.

Customer experience

Responsive storefront, rich categories, search, filters, cart, payments, profile and order tracking.

Business services

Catalog, account, address, order, payment, inventory, wishlist, reviews and subscriber APIs.

Data persistence

SQL Server stores core commerce records; browser storage supports fast cart and user-interface state.

Scope and objectives

Primary objectives

  • Provide a modern multi-category shopping interface.
  • Support desktop, tablet and mobile layouts.
  • Separate frontend and backend responsibilities.
  • Persist users, products, orders, payments and reviews.
  • Demonstrate realistic checkout and account workflows.

Project boundaries

  • No admin dashboard is exposed in the current frontend.
  • Shipping carrier integration is simulated.
  • Email/SMS and production OTP delivery are not connected.
  • Payment verification requires security hardening before real use.
  • Hosting configuration is not yet production-ready.
Overall assessmentThe project is suitable as a substantial portfolio demonstration. It shows frontend composition, responsive design, API integration, relational modelling, checkout logic and legacy-database compatibility. It should be presented as a working full-stack prototype, not yet as a production commerce service.

Request flow

  1. A user interacts with a page or shared component in the Next.js frontend.
  2. The frontend calls apiFetch(path, init) from lib/api.ts.
  3. buildApiUrl combines the path with NEXT_PUBLIC_API_URL.
  4. An ASP.NET controller validates and processes the request.
  5. EF Core reads or changes records in Shopore_DB.
  6. The backend response mapper returns frontend-friendly JSON.
  7. The UI updates its component state and, where applicable, its browser cache.

Separation of concerns

Frontend owns

  • Rendering and navigation
  • Responsive interaction
  • Catalog filters and presentation
  • Cart cache and optimistic wishlist state
  • Payment SDK launch
  • Toasts, loaders and skeletons

Backend owns

  • User and profile persistence
  • Address and order persistence
  • Product catalog and inventory
  • Payment/order records
  • Verified-purchase review eligibility
  • Database compatibility and seed routines
Primary integration seamlib/api.ts is the durable frontend/backend boundary. Changing the API host should require updating NEXT_PUBLIC_API_URL, not editing every page.

05 · Frontend

Frontend design

Shared shell

app/layout.tsx composes the moving offer banner, responsive navigation, page content, footer, toast system, payment scripts and production analytics. The navbar contains department navigation, search, chat support, account access, notifications, bag status and a mobile drawer.

Catalog system

ProductListingPage provides a reusable catalog experience. Individual department route files supply configuration such as title, image directory, brands, sizes and product files. Catalog screens support:

Discovery

Departments, mega menus, landing campaigns and search.

Refinement

Brand, category, price, size, color, discount and rating filters.

Ordering

Popularity, price and other sort choices with responsive controls.

Product details

The product page retrieves backend data when available and accepts route-query fallback data for catalog compatibility. It displays pricing, discount, size selection, delivery PIN validation, product services, bag/wishlist actions and a verified-purchase review section.

Design system

The visual language uses a dark navy primary color, pink accents, light neutral surfaces, rounded cards and restrained shadows. Shared Radix-based primitives provide accessible interaction patterns while Tailwind utilities handle most responsive spacing and layout.

Asset libraryThe public/ directory contains close to one thousand catalog and marketing assets across AVIF, WebP, PNG, JPG, SVG, GIF and MP4 formats. In production, these should be compressed, deduplicated and moved to object storage or a CDN.

State strategy

The project intentionally combines server state with lightweight browser persistence. This makes local demos resilient when backend data is incomplete, but production deployment should move authoritative identity, notification, return and avatar data to secure backend services.

Security implicationlocalStorage and sessionStorage are readable by JavaScript. They are acceptable for this prototype but not ideal for long-lived production authentication. Secure HttpOnly cookies and server-enforced authorization are recommended.

Cross-component updates

Custom browser events such as cart, wishlist, notification and avatar update events allow the navbar and pages to refresh immediately after same-tab storage changes.

08 · Workflows

Core customer workflows

Product discovery to cart

  1. User selects a department or searches.
  2. Listing configuration and backend product data are normalized into cards.
  3. User filters/sorts and opens /products/[id].
  4. User selects size and validates delivery PIN if desired.
  5. addToCart creates a variant-aware key from product ID, image and size.
  6. The bag count updates through a custom browser event.

Checkout and COD order

  1. User reviews cart items and quantities.
  2. User logs in and selects or creates an address.
  3. User selects Cash on Delivery.
  4. Frontend posts the customer, address, total and item snapshots to POST /api/orders.
  5. The backend creates an order record per cart item and returns order DTOs.
  6. Frontend clears the bag and displays order history.

Online payment order

  1. Frontend requests a provider order from POST /api/payment/create-order.
  2. Razorpay or Google Pay is opened on the client.
  3. Client receives payment identifiers/token data.
  4. Frontend posts payment and order data to POST /api/payment/verify.
  5. Backend creates order and payment records together.
Current payment boundaryThe backend currently checks that a payment ID exists but does not cryptographically verify the Razorpay signature. This flow is appropriate for a local demonstration only.

Verified-purchase review

  1. Product review API finds an order item belonging to the user and product.
  2. Cancelled/refunded orders are excluded.
  3. One review is allowed per user and order item.
  4. After saving, aggregate rating and review count are recalculated on the product.

09 · Backend

Backend design

Application startup

Program.cs registers controllers, JSON number handling, Swagger, CORS, EF Core SQL Server and the application middleware. During startup it ensures the database exists and runs compatibility/normalization/seeding services.

Startup services

  • DemoReviewSeeder.SeedAsync

Service utilities

  • PBKDF2 password hashing
  • Random token generation
  • Slug generation
  • Response mapping
  • Product-size catalog behavior

Controller groups

AuthProductsCategoriesStockAddressesProfileOrdersPaymentsWishlistReviewsPIN codeContactChatSubscriber

Legacy database compatibility

The backend is not designed against an empty ideal schema only. It accounts for existing columns and constraints in Shopore_DB. Compatibility services add or normalize required structures at startup. Response mapping and order snapshots also reduce frontend breakage when older records differ from current models.

Strong architectural decisionOrder items store name, brand, image, size, color and price snapshots. Historical order display therefore does not depend entirely on the current product record.

CORS

The configured CORS policy allows local development origins: localhost, 127.0.0.1 and private hosts beginning with 192.168.. Production domains must be explicitly configured before hosting.

12 · Security

Security assessment

Security already present

  • Passwords use PBKDF2-SHA256 with random 16-byte salt, a 32-byte key and 100,000 iterations.
  • Password comparison uses constant-time cryptographic equality.
  • SQL access is performed through EF Core rather than concatenated user input in normal controllers.
  • SQL credentials are intended to be stored using .NET user-secrets for local work.
  • CORS is limited to local/private development hosts.
  • Review eligibility is checked against an owned order item.
  • Address updates/deletes filter by both address and user ID.

Production security gaps

GapCurrent stateRequired production change
Authentication enforcementRandom tokens are returned but not persisted or validated; no authentication middleware is configured.Use ASP.NET Identity or signed JWT/access-refresh tokens; add UseAuthentication and [Authorize].
Object authorizationMany endpoints trust a submitted user_id.Derive user identity from authenticated claims, never from request ownership fields.
Token storageTokens live in local/session storage.Prefer short-lived tokens and secure HttpOnly, SameSite cookies where appropriate.
Payment integrityPayment signature is stored but not validated.Verify Razorpay HMAC server-side; use provider API/webhooks; make order/payment updates idempotent.
OTPOTP is returned in the API response.Send through verified SMS/email provider, store a hash, rate-limit attempts and never return OTP.
Password resetEmail + phone match can reset password.Require a single-use, expiring verified reset token.
Admin operationsProduct/category/stock writes have no role protection.Add admin roles, authorization policies and audit logging.
Rate limitingNot configured.Rate-limit login, OTP, contact, chat, review and payment endpoints.
SecretsConnection-string pattern exists in settings and user-secrets guidance.Remove committed secrets, rotate credentials and use host secret management.
Do not accept real customer payments yetThe current payment endpoints are demonstration code. Real payment processing requires server-side signature verification, webhook verification, idempotency and protected order ownership.

13 · Setup

Local development setup

Prerequisites

  • Node.js compatible with Next.js 16 and npm.
  • .NET 10 SDK.
  • Microsoft SQL Server with access to Shopore_DB.
  • A browser; optional Razorpay test key for payment UI.

1. Configure SQL Server secret

cd D:\Shoporestore\Shoporestore\Shoporestore_Backend
dotnet user-secrets set "ConnectionStrings:ShoporeDb" \
  "Server=YOUR_SERVER;Database=Shopore_DB;User Id=YOUR_USER;Password=YOUR_PASSWORD;TrustServerCertificate=True;MultipleActiveResultSets=True" \
  --project .\Shoporestore.Api\Shoporestore.Api.csproj

2. Start the backend

dotnet run --project .\Shoporestore.Api\Shoporestore.Api.csproj --urls http://localhost:5285

Swagger is available in development at http://localhost:5285/swagger.

3. Configure frontend environment

# .env.local
NEXT_PUBLIC_API_URL=http://localhost:5285
NEXT_PUBLIC_RAZORPAY_KEY_ID=your_test_key_if_used

4. Install and start frontend

cd D:\Shoporestore\Shoporestore
npm install
npm run dev

Open the local URL printed by Next.js, normally http://localhost:3000.

Recommended startup order

  1. Start SQL Server.
  2. Start ASP.NET Core and confirm Swagger/products endpoint.
  3. Start Next.js.
  4. Register a customer and complete a COD smoke test.
Configuration ruleDo not hard-code production URLs in components. Keep all frontend API requests behind NEXT_PUBLIC_API_URL and apiFetch.

15 · Hosting

Deployment plan

The frontend can be hosted independently, but a live full-stack system also needs a reachable ASP.NET Core service and SQL Server database. A frontend-only deployment cannot complete login, profiles, address persistence, orders or reviews.

Vercel / Node host
Next.js frontend
→
Azure App Service / VM
ASP.NET Core API
→
Azure SQL / hosted SQL Server
Private credentials

Deployment sequence

  1. Provision hosted SQL Server and migrate/validate the schema.
  2. Create a production SQL login with minimum permissions.
  3. Deploy the backend with production connection string in host secrets.
  4. Configure HTTPS, allowed origins, logging, health checks and backups.
  5. Set NEXT_PUBLIC_API_URL to the public HTTPS API URL.
  6. Deploy the frontend and run end-to-end smoke tests.
  7. Configure real payment credentials only after verification logic is hardened.

Free/low-cost reality

Frontend hosting commonly has free tiers. Persistent ASP.NET hosting and managed SQL Server are harder to keep completely free and may sleep, expire or limit database size. For interviews, a local demonstration plus repository and this architecture document is acceptable if you clearly explain that hosting cost and production security work remain.

Interview-safe explanation

Suggested answer“The complete project runs locally with Next.js, ASP.NET Core and SQL Server. I have not exposed the database or prototype payment/authentication flow publicly because I would first add production authentication, payment verification, secrets management and a managed SQL environment. I can demonstrate the entire flow locally and explain the deployment architecture.”

17 · Portfolio

How to present this project in an interview

60-second summary

“Shopore is a responsive full-stack e-commerce application. I built the customer storefront with Next.js, React and TypeScript, then separated the business API into ASP.NET Core with EF Core and SQL Server. It covers catalog filtering, product details, cart, wishlist, addresses, COD/online checkout, orders, cancellation and verified-buyer reviews. A key engineering challenge was making the new API work with an existing legacy SQL Server schema while keeping the frontend contract stable.”

Best technical talking points

  • Architecture: shared apiFetch bridge decouples frontend deployment from backend location.
  • Data integrity: order item snapshots preserve purchase history.
  • Legacy compatibility: startup schema services and response mapping support the existing database.
  • User experience: consistent loaders, responsive grids, mobile drawers and touch-oriented controls.
  • Review trust: backend checks an owned order item before allowing a review.
  • Honest engineering judgment: identify security gaps and explain the exact production path.

Suggested demonstration order

  1. Open responsive landing page and mobile navbar.
  2. Browse a department; demonstrate filters and sorting.
  3. Open a product, select size, verify PIN and add to bag.
  4. Login/register and manage an address.
  5. Place a COD order to avoid presenting prototype online payment as production-ready.
  6. Open My Orders and show order detail/cancellation.
  7. Show Swagger and explain SQL Server entities.
  8. Finish with the security/hosting roadmap.

Questions you should be ready for

QuestionStrong answer direction
Why Next.js with ASP.NET Core?Next.js provides a rich React storefront; ASP.NET offers a typed API and strong SQL Server/EF Core integration.
Why localStorage for cart?It provides instant guest persistence; production can merge it into a server cart after login.

Troubleshooting guide

SymptomLikely causeAction
Frontend says it cannot connectAPI is stopped or NEXT_PUBLIC_API_URL is wrong.Start port 5285 backend; restart Next.js after environment changes.
CORS errorFrontend host is outside allowed local origins.Add the exact production/development origin to backend CORS policy.
Backend fails at startupMissing/invalid connection string or SQL Server unavailable.Check user-secrets, network, SQL login and database name.
Invalid column / constraint errorLegacy schema differs from current entity assumptions.Inspect live schema and update compatibility service before resetting data.
Old product data in orderExpected snapshot behavior.Order history intentionally shows purchase-time details.
Wishlist returns product unavailableLocal catalog product ID does not exist in backend Products.Seed/match the backend product or use backend-resolved product IDs.
Razorpay key missingEnvironment variable absent.Add test key to .env.local and restart frontend.
Order loading times outBackend/SQL request exceeds the three-second UI budget.Check API logs, database latency and indexes; UI keeps previous data when available.
Mobile menu overlays incorrectlyStale dev bundle or viewport not refreshed.Restart dev server, clear build cache if necessary, and retest at 320–640px.

Do not “fix” legacy errors by deleting data first

The project deliberately uses a legacy database. Inspect the schema, constraint and payload mismatch before rebuilding tables. Compatibility changes are safer than resets when existing user/order data matters.
